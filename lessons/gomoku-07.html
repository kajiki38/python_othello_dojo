発展: 2手先を読むAI

Lesson 6 で作った ai_strong は、「ダブルリーチ」という五目並べ専用テクニックの塊でした。発展編の今回は、オセロでも将棋でもチェスでも通用する汎用アイデア——相手の返し手まで読むミニマックス——を武器に、あの王者に挑みます。

🎯 このレッスンのゴール

  • 「相手は一番イヤな返しをしてくる」と仮定して読むミニマックスの考え方を知る
  • 盤面を点数にする評価関数と、読む手を絞る候補リストを作る
  • 再帰(自分を呼ぶ関数)で2手先を読む ai_minimax を完成させる

STEP 1「1手先」の限界 — 三々の罠

ai_strong の読みは、実は3種類だけです。「今すぐ勝てる手」「今すぐ止めるべき手」、そして「1手でできる(できてしまう)ダブルリーチ」。逆に言えば、2手かけて仕込まれる罠は見えません。その代表が、縦横2方向に同時に3連を作る「三々(さんさん)」です。

白の番です。ai_strong は★の危険に気づけるでしょうか。実行して確かめましょう。

白は中央の (4, 4) を選びました。一見悪くない手ですが、★は放置です。黒が (5, 3) に置くと、「次に置けばダブルリーチになる点」が4つも出現します。白が1手で守れるのは1つだけ——実はこの時点で、どう受けても黒の勝ちが確定しています。ai_strong のセンサーは「1手でできるダブルリーチ」までしか届かないので、この仕込みが見えないのです。

いたちごっこの予感: では「三々センサー」を追加しますか?その次は「四三」、その次は…と、パターンを覚えさせるやり方はきりがありません。今回は発想を変えて、「相手の返し手まで自分で読む」汎用の仕組みを作ります。これならパターンを1つも教えなくても、読みの深さのぶんだけ罠を見抜けるようになります。

STEP 2相手は一番イヤな返しをしてくる — 盤面を点数にする

先読みの心臓部は、たった1つの仮定です。「自分がこの手を打ったら、相手は相手にとって最善(=こちらに一番イヤ)な返しをしてくる」。だから、ある手の本当の価値は「打った直後の盤面」ではなく、相手が最善の返しをした後の盤面で測るべきなのです。

max と min: 自分の番では、いちばん点数の高い手を選ぶ(max)。相手の番では、こちらの点数がいちばん低くなる手を選ばれる(min)。この max と min を交互に繰り返して読む方法をミニマックスと呼びます。名前そのまんまですね。

ミニマックスには「盤面の点数」が必要です。そこで、盤面を受け取って数値を返す関数=評価関数を作ります。ゲームAIの心臓部で、設計は自由。今回は Lesson 6 で鍛えた道具 count_reaches をそのまま使い、「リーチ数の差」を点数にします。ダブルリーチは「片方しか止められない=ほぼ勝ち」でしたから、特別に大きな点をつけます。

リーチ1つで 10 点、ダブルリーチで 500 点。同じ盤面でも color を変えると符号が反転するのがポイントです(黒に良い盤面は白には悪い)。「誰の視点で採点するか」を引数で切り替えられるおかげで、この後のミニマックスでは常に自分視点の点数だけを考えればよくなります。

この evaluate、3連が 0 点だったりと、かなり素朴です。でも心配いりません。素朴な評価関数でも、読みと組み合わせると何倍も賢く見える——それをこれから確かめます。

STEP 381マス全部は読めない — 候補を絞る

2手先を読むとは「自分の手×相手の返し」を総当たりすることです。全マスを候補にすると、序盤でも 79 × 78 ≒ 6000 通り、3手先なら約47万通り。組み合わせは深さのぶんだけ掛け算で爆発します。でも Lesson 6 で学んだとおり、意味のある手はほぼ「石の近く」だけ。そこで near_cells を中央に近い順に並べ、上位8個だけを読む候補にします。

6006 通りが 64 通りに——約100分の1です。「読む手を絞る」のは、将棋やチェスの本格AIでも使われている常套手段です。もちろん絞りすぎると好手を見落とすので、速さと強さのトレードオフになります。8個はそのバランスの落としどころです(チャレンジでいじってみましょう)。

STEP 4再帰 — 関数が自分を呼ぶ

いよいよ「2手先を読む」を関数にします。カギになる発想はこうです。「depth 手先まで読む」は「1手進めて、残り depth−1 手先まで読む」と同じ——つまり、関数の中で自分自身を呼べばいい。まずはこの「自分を呼ぶ関数」を、いちばん小さな例で体験しましょう。

関数が自分自身を呼ぶ書き方を再帰といいます。①必ず「止まる条件」を先に書く ②呼ぶたびに問題を小さくする(n - 1、depth - 1)——この2点さえ守れば怖くありません。ミニマックスは『depth 手先まで読む』を『1手進めて depth−1 手先まで読む』に置き換える、再帰の代表例です。

では本番、minimax_score です。引数は4つ: board(盤面)、color(誰の視点で採点するか。読みの間ずっと固定)、turn(いま打つ番)、depth(あと何手読むか)。depth == 0 が止まる条件で、読みの底に着いたら evaluate で点数化。途中で今打つ側に即勝ち手があれば、そこで ±1000 点の決着です(ダブルリーチの500点より上——「確定勝ち」は「ほぼ勝ち」より偉い)。それ以外は候補を1手ずつ試して再帰し、自分の番なら max、相手の番なら min を取ります。

3連の端 (1, 4) や (5, 4) を止めれば -10 点で済むのに、関係ない (0, 0) に置くと -500 点。「黒が open な4連を作ってくる未来」まで織り込んだ点数になっています。これが「相手は一番イヤな返しをしてくる、と仮定して測る」ということです。

候補に打つとき、board[y][x] = turn で直接書き換えて、再帰から戻ったら必ず EMPTY に戻す方式にしています。「試し置き→戻す」は Lesson 5〜6 でおなじみの型ですね。オセロだと石がひっくり返るので盤のコピーが必要ですが、五目並べは置くだけなので、この方式が使えてコピーのぶん速くなります。

STEP 5ai_minimax 完成 — 王者 ai_strong との対戦実験

仕上げは簡単です。自分の候補それぞれに minimax_score で点数をつけ、いちばん高い手を選ぶだけ。ただし「即勝ち」と「相手の即勝ちのブロック」だけは、読みに入る前に即断します(速くて確実なので)。まずは STEP 4 と同じ「両端の空いた3連」で試してみましょう。

白は (5, 4) — 3連の端です。注目してほしいのは、このAIには「3連を止めろ」とも「ダブルリーチ」ともひと言も教えていないこと。evaluate と2手の読みだけから、防御が勝手に生まれたのです。パターンを教える代わりに読みを教える——これがミニマックスの威力です。

では対戦実験です。前回の ai_block と10局、王者 ai_strong と10局(いずれも先後交代)。※読みが入るぶん、十数秒かかることがあります。

筆者の実測(各50局)では、vs ai_block は50勝0敗、vs ai_strong は8勝33敗9分でした。正直に言いましょう——ふつうの相手には全勝級ですが、専用テクを磨き込んだ王者はまだ手強い。それでも、五目並べのルール以外ほとんど何も教えていない汎用アイデア1本でここまで戦えたこと自体がすごいのです。しかも1手約12ミリ秒と、まだ余力たっぷり。王者超えは、この下のチャレンジで狙います。

チャレンジ:改造ミニマックスで ai_strong に挑め

チューニングで王者に挑みます。改造ポイントは3つ。① evaluate2 — リーチの重みを攻め寄りに変える(例: 自分×15・相手×12。攻め続ければ相手に形を作る暇を与えません)。② candidates2 — 候補を増やす(例: 10個。見落としが減るぶん少し遅くなります)。③ ai_minimax2 — ai_minimax と同じ「勝ち手 → ブロック → ミニマックス」の形に完成させる(いまは2つのチェックが抜けています)。実行すると複数の盤面での自動チェックに加えて、ai_strong との10局対戦で勝率も出ます(十数秒かかります)。勝ち越せたら勝利宣言!

解答例を見る 
def evaluate2(board, color):
    my = count_reaches(board, color)
    op = count_reaches(board, opponent(color))
    if my >= 2:
        return 500
    if op >= 2:
        return -500
    return my * 15 - op * 12   # 攻め寄りの重みに変更

def candidates2(board):
    cand = near_cells(board)
    if not cand:
        return [(SIZE // 2, SIZE // 2)]
    center = SIZE // 2
    cand.sort(key=lambda m: (m[0] - center) ** 2 + (m[1] - center) ** 2)
    return cand[:10]           # 候補を10個に拡大

def minimax_score2(board, color, turn, depth):
    if depth == 0:
        return evaluate2(board, color)
    win = find_winning_move(board, turn, near_cells(board))
    if win:
        return 1000 if turn == color else -1000
    scores = []
    for x, y in candidates2(board):
        board[y][x] = turn
        scores.append(minimax_score2(board, color, opponent(turn), depth - 1))
        board[y][x] = EMPTY
    if not scores:
        return evaluate2(board, color)
    if turn == color:
        return max(scores)
    return min(scores)

def ai_minimax2(board, color, moves, depth=2):
    win_move = find_winning_move(board, color, moves)
    if win_move:
        return win_move
    block_move = find_winning_move(board, opponent(color), moves)
    if block_move:
        return block_move
    best_move, best_score = None, None
    for x, y in candidates2(board):
        board[y][x] = color
        s = minimax_score2(board, color, opponent(color), depth - 1)
        board[y][x] = EMPTY
        if best_score is None or s > best_score:
            best_move, best_score = (x, y), s
    return best_move

# ai_strong と10局対戦(先後交代)。勝ち越せたら勝利宣言!
wins = losses = draws = 0
for i in range(10):
    if i % 2 == 0:
        winner = play_match(ai_minimax2, ai_strong)
        my_color = BLACK
    else:
        winner = play_match(ai_strong, ai_minimax2)
        my_color = WHITE
    if winner == my_color:
        wins += 1
    elif winner is None:
        draws += 1
    else:
        losses += 1
print(f"ai_minimax2 vs ai_strong: {wins}勝 {losses}敗 {draws}分")

筆者の実測(50局)では、この設定で 7勝23敗20分。元の ai_minimax(8勝33敗9分)と比べて、負けが33→23に減り、引き分けが9→20に増えました——まず「負けない碁」に近づいた形です。さらに攻めを強めて my * 20 - op * 8 にすると 15勝20敗15分 と、ほぼ互角まで迫れました。10局勝負は結果がかなりブレるので、何度か実行して流れをつかみつつ、重みや候補数を自分の手で動かして勝ち越しを狙ってみてください。ちなみに手元では depth=3 に上げると逆に負けが増えました。深く読めば必ず強くなるとは限らない——評価関数とのバランスがある——のもミニマックスの奥深さです。この「仮説→実験→測定」のループこそ、AI開発のいちばんリアルな姿です。

STEP 6まとめ — どのゲームでも同じ形

  • ミニマックスは「相手は一番イヤな返しをしてくる」前提の読み。自分の番は max、相手の番は min
  • 読みの底では評価関数で盤面を点数化する。素朴な点数でも、読みと組み合わせると開花する
  • 再帰は「止まる条件+問題を小さくして自分を呼ぶ」。depth がカウントダウンの役目
  • 候補絞りで読む量をコントロールする。候補の数×深さが速さと強さを決める

そして最大の収穫はこれです——evaluate と candidates を差し替えれば、この形はオセロでもコネクトフォーでも、将棋やチェスでもそのまま通用します。ダブルリーチのような「種目専用の知識」と、ミニマックスのような「汎用の読み」。その両方を自分の手で書いたあなたは、もうゲームAI作りの入り口を堂々と通過しています。おつかれさまでした!
